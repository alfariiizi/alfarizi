# Resume Page and PDF Generation Design

## Objective

Create a dedicated `/resume` experience that serves three related outputs from the same resume data:

- a web page at `/resume`
- an ATS-friendly PDF download
- an original, richly styled PDF download

The page should present a resume-style summary with a download header at the top and a rendered preview section below. The downloadable PDFs should be generated by the project itself during build or deployment, not hand-authored static files.

## Approved Direction

The approved direction for this work is:

- keep one resume source of truth that powers both the `/resume` page and the generated PDFs
- reuse existing site data where it already exists, especially projects from Velite and experience from the existing experience data
- generate PDFs from HTML and Tailwind styling using a headless browser print workflow
- make the `ATS Friendly` PDF intentionally simple and semantic
- make the `Original` PDF visually richer and fully Tailwind-driven
- use a build/deploy timestamp for the generated PDF filenames
- keep the timestamp stable for a given build so the page links and generated files always match

## Constraints

- Do not create duplicate resume content in multiple places.
- Do not introduce a separate CMS or external data source for this feature.
- Do not make the PDF generation depend on manual post-deploy steps.
- Do not use `react-pdf` as the primary rendering path for this feature.
- Do not weaken the existing visual language of the site outside the resume area.
- Keep the generated filenames filesystem-safe.
- Keep the public download URLs stable for a single build.

## Research Summary

Two approaches were considered for PDF generation:

- `react-pdf` and Tailwind adapters such as `react-pdf-tailwind`
- browser-based printing from rendered HTML and Tailwind

The browser-print approach is the selected one.

Reasons:

- `react-pdf` uses its own layout/styling model rather than the normal Tailwind web pipeline
- Tailwind adapters for `react-pdf` support only a subset of Tailwind-compatible styling
- the `Original` PDF requirement needs the same high-fidelity Tailwind styling as the web preview
- browser print keeps the HTML, CSS, and PDF preview aligned

The selected runtime should be a headless Chromium flow through Puppeteer, because the PDF workflow only needs `page.pdf()` and a minimal build-time browser automation surface.

## Data Model

### Resume Source of Truth

Add a dedicated resume data module that composes existing site data into a single resume-shaped object.

That object should contain:

- identity
- contact details
- summary
- experience entries
- selected projects
- skills or capabilities
- optional education or additional notes if the existing content supports it

### Data Reuse Rules

- Experience should come from the existing `src/app/experience/data.ts` source.
- Projects should come from Velite project data, preferably via the existing highlighted project selection.
- Social/contact links should reuse the existing site data where possible.
- Resume-specific wording should live in the resume data module, not scattered across page components.

### Selection Rules

- Prefer a small number of selected projects rather than dumping the full portfolio into the resume.
- Use the strongest and most representative projects already present in the portfolio data.
- Keep the resume concise enough to read like a real resume, not a full portfolio archive.

## Page Structure

### `/resume` Page

The page should have two major sections:

1. `Download Resume`
2. `Rendered Preview`

### Download Resume Section

This section should contain two links:

- `ATS Friendly` -> `/resume/ats/{build-timestamp} - Moh Rizal Alfarizi.pdf`
- `Original` -> `/resume/ori/{build-timestamp} - Moh Rizal Alfarizi.pdf`

The timestamp must be generated once per build/deploy and reused everywhere the resume links or filenames are referenced.

### Rendered Preview Section

This section should show the resume output below the download header.

Recommended presentation:

- use tabs or a similar toggle to switch between `ATS Friendly` and `Original`
- render each variant using the same underlying data
- keep the preview close to the final PDF layout

The preview should be readable without requiring any download action.

## PDF Generation Pipeline

### Build-Time Generation

PDF files should be generated during the repo build or deployment pipeline.

The build pipeline should:

1. create a single build timestamp
2. write that timestamp into a build artifact or generated source file that both the app and PDF generator can import
3. build the Next.js app
4. start the built app locally
5. open the printable resume routes in a headless browser
6. save the generated PDFs into `public/resume/ats/` and `public/resume/ori/`

### File Naming

Use a filesystem-safe timestamp format, for example:

- `2026-06-13 14-05 - Moh Rizal Alfarizi.pdf`

The exact formatter can be finalized in implementation, but it must:

- be stable for the entire build
- avoid characters that are unsafe on common filesystems
- remain human-readable

### Printable Routes

Add internal printable routes for generating the PDFs.

These routes should:

- reuse the same resume data
- be hidden from normal navigation
- be marked `noindex`
- avoid interactive UI that should not appear in a PDF

## Layout Rules

### ATS Friendly Variant

The ATS variant should be minimal and semantically structured.

It should prioritize:

- one-column flow
- strong heading hierarchy
- plain text
- clean spacing
- minimal decoration
- maximum text extractability

It should avoid:

- decorative shapes
- unnecessary background treatments
- layout tricks that do not help ATS readability

### Original Variant

The original variant should remain Tailwind-forward and visually richer.

It should still be resume-like, not poster-like.

It may include:

- stronger typographic hierarchy
- restrained color accents
- section framing
- denser but still readable layout

It should avoid:

- animations
- interactive controls inside the PDF output
- layout elements that break print stability

## Rendering Rules

- Wait for fonts to load before printing.
- Use print-safe spacing and page-break controls where needed.
- Preserve color fidelity in the original variant.
- Keep page breaks from cutting through critical sections whenever practical.
- Ensure the printable routes are stable when rendered headlessly.

## Route and Asset Layout

Recommended structure:

- `src/app/resume/page.tsx`
- `src/app/resume/data.ts`
- `src/app/resume/_components/*`
- internal printable routes under `src/app/resume/print/*`
- generated PDFs in `public/resume/ats/`
- generated PDFs in `public/resume/ori/`
- generated build stamp in a repo-local source file or equivalent build artifact

The exact file names can vary slightly, but the boundaries should remain the same:

- one shared data layer
- one public preview page
- one printable layer
- one build-time PDF output location

## Integration With Existing Site Data

### Projects

Use Velite project data instead of recreating project records for resume purposes.

The resume should surface only the selected projects that best support the story of the work.

### Experience

Reuse the existing experience entries rather than duplicating them.

If the resume needs shorter wording than the experience page, derive a resume-specific view model from the same source.

### Contact

Reuse the current site contact and social information where available.

If a field is not already present in the repo, add it to the resume source of truth rather than scattering it into component props.

## Verification

The implementation is successful if:

- `/resume` renders with the download header and preview section
- the generated PDF filenames match the build timestamp used in the page links
- both PDF variants are generated during build or deployment
- the ATS file is readable as a plain resume
- the original file keeps Tailwind styling intact
- the project does not rely on manual PDF export steps

## Non-Goals

- Do not redesign the whole site around the resume page.
- Do not convert the rest of the site to print-specific behavior.
- Do not add a generic document builder.
- Do not add an external resume CMS.
